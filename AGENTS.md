## Project Overview

URGE (Universal Ruby Game Engine) is a 2D/3D game engine compatible with the RGSS 1/2/3 API, written in C++20. It embeds an MRI Ruby interpreter for game scripts, uses SDL3 for windowing/input/audio, and DiligentCore as the rendering RHI (D3D12/Vulkan/OpenGL/Metal). Targets Windows, Linux, macOS, Android, and Emscripten/WASM.

The codebase style is heavily inspired by The Chromium Project — same layering (`base/`, `content/`, `components/`, `ui/`), same primitives (`base::RefCounted`, `base::WeakPtr`, `scoped_refptr`, `ExecutionContext`), same naming conventions.

## Build & Run

The repository must be cloned with submodules — `third_party/` and several nested third-party directories are git submodules:

```bash
git clone --recursive https://github.com/Admenri/urge.git
# or, if already cloned:
git submodule update --init --recursive
```

Configure and build (out-of-tree build directory is `out/` or `build/`):

```bash
cmake -S . -B out
cmake --build out --target Game     # Windows / Linux / macOS executable
# Android: target is `main` (shared library)
# Emscripten: target is `index` (produces index.html)
```

Required toolchain: CMake ≥ 3.21, Python ≥ 3, Ruby ≥ 1.8 (non-Windows; Windows uses bundled Ruby in `binding/mri/third_party/ruby`), Vulkan SDK ≥ 1.1, NASM ≥ 2.14 on Windows, MSVC ≥ 2015 with Clang/GNU CLI on Windows, optional Ninja.

The runtime expects an INI configuration file alongside the binary (e.g. `Game.ini`). A reference is at [doc/engine_config.ini](doc/engine_config.ini); i18n strings at [doc/engine_i18n.xml](doc/engine_i18n.xml). The executable uses its own filename (without extension) to derive the INI path.

`/WX` (MSVC) and `-Werror` (others) are set on `engine_content` — warnings are build failures there.

## Code Generation Pipeline (Critical)

The Ruby ↔ C++ binding layer is **autogenerated at CMake configure time**. Do not hand-edit generated files.

- **Source of truth:** [content/public/*.h](content/public/) — public-facing engine interfaces marked with the `URGE_OBJECT(name)` macro and special `/*--urge(...)--*/` comments above each class/method/property.
- **Generator:** [binding/mri/tools/autogen.py](binding/mri/tools/autogen.py) drives [binding/mri/tools/bindgen.py](binding/mri/tools/bindgen.py), which uses [binding/build/api_parser.py](binding/build/api_parser.py) to parse the `/*--urge(...)--*/` annotations and emit Ruby C bindings.
- **Output:** `${CMAKE_CURRENT_BINARY_DIR}/gen/binding/mri/autogen_*_binding.{h,cc}` — these are added to the `binding_mri` static library.
- **Cache invalidation:** `doc/api/api_hash.txt` stores an MD5 of the API headers; regeneration only runs when this hash changes.

When adding/modifying a public API:
1. Edit the relevant header in [content/public/](content/public/) (use `URGE_OBJECT`, `/*--urge(name:foo)--*/` annotations, and the macros in [content/content_config.h](content/content_config.h) — `URGE_EXPORT_ATTRIBUTE`, `URGE_EXPORT_DISPOSABLE`, `URGE_EXPORT_SERIALIZABLE`, etc.).
2. Implement it in the matching `*_impl.{cc,h}` under [content/canvas/](content/canvas/), [content/render/](content/render/), [content/screen/](content/screen/), [content/gpu/](content/gpu/), etc.
3. Re-run CMake configure to regenerate bindings (or rebuild — CMake `execute_process` runs the generator each configure).

The MD5 hash check means simply rebuilding without re-configuring will *not* re-run the generator if you only edit binding-irrelevant code, but any change to a `content/public/*.h` file requires a CMake configure step to refresh the bindings.

## Architecture

The build graph (`app/CMakeLists.txt`) links the executable against five libraries:

```
Game ──┬── engine_base       (base/)         — Chromium-style primitives: RefCounted, WeakPtr, Bind/Callback, threading, math
       ├── engine_ui         (ui/widget/)    — SDL3 window wrapper (ui::Widget)
       ├── engine_renderer   (renderer/)     — DiligentCore wrapper: device, pipeline, layout, resource
       ├── engine_content    (content/)      — Engine "DOM": all public API impls + main loop runner
       └── binding_mri       (binding/mri/)  — Ruby MRI integration + autogenerated bindings
```

Plus standalone components linked in via `engine_content`: [components/filesystem/](components/filesystem/) (physfs-backed `IOService`), [components/audioservice/](components/audioservice/) (miniaudio), [components/fpslimiter/](components/fpslimiter/), [components/network/](components/network/) (asio + websocketpp, not on Emscripten where a JS/fetch-based impl is used), [components/version/](components/version/).

### Boot flow ([app/urge_main.cc](app/urge_main.cc))

1. Parse `argv[0]` → derive app name + INI path (`Game.exe` → `Game.ini`).
2. Build `filesystem::IOService` (physfs mount of CWD).
3. Load `ContentProfile` from INI; apply CLI overrides.
4. Init spdlog logger (`base::logging`), SDL, SDL_ttf.
5. Load i18n XML and the default font into `ScopedFontData`.
6. Create the `ui::Widget` window from profile params (size, fullscreen, GL/Vulkan).
7. Construct `content::ContentRunner` with a `binding::BindingEngineMri` entry.
8. `runner->RunMainLoop()` blocks until exit/reset.

### Content layer (`content/`)

- [content/context/](content/context/) — `ExecutionContext` is the central per-runner context object passed everywhere; `ExceptionState` for error propagation; `Disposable` mixin; `EngineObject` base for objects bound to a context.
- [content/public/](content/public/) — annotated public API headers (the binding generator's input). One header per exposed type (`engine_sprite.h`, `engine_bitmap.h`, `engine_audio.h`, `engine_gpu*.h`, …). The `URGE_OBJECT(T)` macro expands so `T` publicly inherits `base::RefCounted<T>`.
- `*_impl.{cc,h}` — the actual implementations, organized by subsystem: `canvas/` (Bitmap/Surface/Font/Animation), `render/` (Sprite/Plane/Tilemap/Window/VideoDecoder + drawable controller + sprite_batch + pipeline_collection), `screen/` (RenderScreen/Viewport/Display), `gpu/` (low-level GPU wrappers around DiligentCore), `media/` (Audio/AudioStream), `input/` (keyboard/mouse/touch/text), `io/` (IOStream), `network/`, `platform/` (URGE module impl).
- [content/worker/](content/worker/) — `ContentRunner` (tick loop, ImGui dev overlay, FPS monitor); `EngineBindingBase` (script engine integration interface — `BindingEngineMri` implements it); `EventController`.

### Renderer (`renderer/`)

Thin wrapper over DiligentCore: `device/render_device.cc` initializes the RHI; `pipeline/` builds shaders (HLSL — see `builtin_hlsl.cc`) and pipeline state objects; `layout/` describes vertex layouts; `resource/` and `utils/` contain helpers.

### Binding (`binding/mri/`)

- [binding/mri/mri_main.cc](binding/mri/mri_main.cc) (`BindingEngineMri`) implements `EngineBindingBase`: starts the MRI Ruby VM, calls each generated `Init<T>Binding()`, loads the user's packed RGSS scripts, and dispatches main-loop events.
- [binding/mri/mri_util.{cc,h}](binding/mri/mri_util.h) — argument unpacking, type wrappers, `MRI_DECLARE_DATATYPE` machinery.
- [binding/mri/binding_patch.cc](binding/mri/binding_patch.cc) — runtime patches to the MRI VM for the embedded environment.
- `third_party/rbffi`, `third_party/zlib` — ext gems compiled into the binding.

## Conventions

- **C++20** project-wide. `set(CMAKE_CXX_STANDARD 20)` is enforced at the top level.
- **No raw `new` for refcounted types** — use `base::MakeRefCounted<T>()`. `scoped_refptr<T>` for ownership, `T*` raw for non-owning, `base::WeakPtr<T>` to break cycles. This mirrors the `//base` patterns in Chromium.
- **Public API headers go in [content/public/](content/public/) only.** Anything outside that directory is internal and should not be exposed via the `/*--urge(...)--*/` annotations.
- **The `ExceptionState&` parameter is mandatory** on every binding-exposed method — it is the binding ABI for raising Ruby exceptions from C++.
- **Platform branching:** `OS_WIN`, `OS_LINUX`, `OS_ANDROID`, `OS_EMSCRIPTEN`, `OS_MAC` (defined in `base/buildflags/build.h`). For source-level CMake branching, `CMAKE_SYSTEM_NAME` is used (`Windows` / `Linux` / `Darwin` / `Android` / `Emscripten`).
- **Asio is standalone** (`ASIO_STANDALONE=1`) — never include Boost.Asio.
